# Browser QA — Визуальное и функциональное тестирование криптокошелька

## Когда использовать

- После деплоя фичи на staging — быстрая проверка до ручного прогона
- При ревью PR, затрагивающих frontend криптокошелька
- Для E2E-проверки критических флоу: send, receive, биржа
- Для визуальной регрессии после UI-изменений
- Когда нужно убедиться что новая фича не сломала существующее

## Инструмент

Claude in Chrome (Desktop) — основной инструмент команды.  
Тестирование только в **Chrome Desktop**.

---

## Фаза 1: Smoke (5–10 мин) — запускать при каждом деплое

Открыть staging URL криптокошелька
Проверить консоль: нет критических ошибок (JS errors, failed imports)
Игнорировать: аналитика, third-party трекеры
Проверить сеть: нет 4xx / 5xx на ключевых API-запросах
Особое внимание: /balance, /transactions, /rates, /exchange
Скриншот главного экрана (1440px desktop)
Проверить Core Web Vitals: LCP < 2.5s, CLS < 0.1, INP < 200ms


**Крипто-специфика:**
- Баланс подгружается асинхронно — дождаться завершения загрузки, не считать skeleton ошибкой
- Котировки могут запаздывать — проверить что лоадер отображается, а не пустое поле

---

## Фаза 2: Interaction Test — критические флоу кошелька

Auth flow

Открыть приложение неавторизованным → редирект на login
Авторизоваться → попасть на главный экран с балансом
Выйти → редирект на login, сессия очищена


Send (отправка крипты)

Открыть форму Send
Ввести валидный адрес и сумму → кнопка Confirm активна
Ввести сумму больше баланса → ошибка "Insufficient funds"
Ввести невалидный адрес → inline-ошибка валидации
Подтвердить отправку → статус Pending отображается


Receive (получение)

Открыть экран Receive
Адрес кошелька отображается полностью
Кнопка Copy работает (clipboard)
QR-код отображается (если есть)


История транзакций

Список загружается без ошибок
Статусы: Confirmed / Pending / Failed — визуально различимы
Клик на транзакцию открывает детали


Биржа

Переход на биржу работает
Котировки / пары загружаются
При недоступности биржи: error state, не пустой экран




---

## Фаза 3: Visual Regression (только Desktop 1440px)

Скриншоты ключевых страниц:

Главный экран (с балансом и без / empty state)
Send форма (дефолт / с ошибкой)
История транзакций
Экран биржи


Сравнить с Figma-макетом (если есть ссылка в тикете):

Расположение элементов
Состояния: default / hover / disabled / error / loading
Типографика, цвета, отступы


Флаги для немедленного репорта:

Overflow / обрезка текста
Элементы вне экрана
Сломанные иконки / изображения
Несоответствие состоянию загрузки (нет skeleton там где должен быть)




**Крипто-специфика:**
- Проверить отображение длинных адресов (не обрезаются ли криво)
- Проверить большие числа (999999.99999999 BTC — не ломают ли layout)
- Проверить суммы с максимальным числом знаков после запятой

---

## Фаза 4: Edge Cases — состояния данных
Проверить UI при:

Нулевом балансе (0.00)
Пустой истории транзакций (empty state)
Очень большом балансе (layout не ломается)
Pending транзакции в списке
Недоступном API (network error state)
Медленной загрузке (skeleton/spinner виден)

## Формат отчёта
Browser QA Report — [URL] — [дата]
Среда: Chrome Desktop, Staging
Тестировщик: [имя]
Smoke

Консоль: 0 критических ошибок, X warnings (аналитика — игнор)
Сеть: все /balance, /transactions, /rates — 200 ✓
Core Web Vitals: LCP 1.4s ✓, CLS 0.01 ✓, INP 95ms ✓

Interactions

[✓] Auth: вход / выход работает
[✓] Send: happy path, insufficient funds, невалидный адрес
[✗] Send: при вводе суммы 0.00000001 BTC кнопка Confirm остаётся неактивной (баг)
[✓] Receive: адрес, copy, QR
[✓] История: загружается, статусы различимы
[✗] Биржа: при недоступности API — пустой экран вместо error state (баг)

Visual

[✓] Главный экран соответствует Figma
[✗] Send форма: длинный адрес обрезается некорректно на 1440px

Edge Cases

[✓] Empty state (нет транзакций) — корректен
[✗] Pending транзакция — иконка статуса не отображается

Вердикт: БЛОКИРУЮЩИХ — 0 / БАГОВ — 3
Рекомендация: завести баг-репорты по Send (минимальная сумма) и Exchange (error state)

## Связь с основным флоу

Browser QA запускается **после** того как тест-кейсы в Allure написаны и задеплоен билд на staging. Найденные баги оформлять по шаблону из `qa-crypto-supplement`.
Jira тикет → Allure ТК → Деплой на Staging
→ browser-qa (этот скилл)
→ Баг-репорты по шаблону qa-crypto-supplement
